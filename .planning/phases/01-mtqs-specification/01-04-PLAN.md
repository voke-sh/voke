    Create `spec/SCOPE.md` as a single-page normative boundary statement with these FIVE sections (verbatim structure from 01-RESEARCH.md "SCOPE.md Content"):

    # MTQS L1 — Scope

    ## 1. What MTQS L1 Is
    - A deterministic, rule-based static analysis of an MCP tool surface.
    - Runs locally and in CI.
    - Consumes: the `tools/list` surface (name, title, description, inputSchema, outputSchema, annotations).
    - Produces: per-rule findings + per-tool score + server score + A–F tier.

    ## 2. What MTQS L1 Is Not (Hard Boundaries)
    - NOT an LLM-as-judge: no model call in any rule function; rule functions are typed `(ctx: RuleContext) => Finding[]` — pure, synchronous, no IO. (Must contain the literal phrases "no LLM" and "LLM-as-judge".)
    - NOT a gateway or proxy: Voke is a read-only observer of `tools/list`; it NEVER sits in the execution path of tool calls. (Must contain the literal words "gateway" and "proxy".)
    - NOT a runtime monitor: no scheduling, no alerting, no health checks (L3 scope).
    - NOT an agent evaluator: no task execution, no agent loop, no "would an agent succeed?" evaluation (L4 scope).
    - NOT a semantic linter: rules check mechanical properties only; clarity/accuracy judgments require L4 eval.
    - NOT an L2 diff / breaking-change gate (the L1 data model leaves room via contentHash + snapshots, but diffing is not built here). (Must contain the literal token "L2".)

    ## 3. The Determinism Guarantee
    - Same `tools/list` input → same output, every run, on any platform.
    - Requires: no wall-clock references in scoring, sorted tool arrays, integer-first arithmetic, no external network calls in rule functions.
    - Any rule requiring IO, randomness, or a model call MUST NOT be added to L1; it belongs in L4.

    ## 4. Scope Creep Prevention
    - A rule PR MUST cite a primary source (MCP spec / Anthropic guidance / JSON Schema / academic paper) — never Glama.
    - A rule PR MUST demonstrate the check is mechanical (no model in the loop).
    - Any proposed rule requiring semantic understanding → rejected from L1, labeled an L4 candidate.

    ## 5. Version Boundary
    - MTQS v0.1 covers 20 rules across 5 active dimensions (schema, description, naming, parameters, annotations).
    - v0.2 adds 16 differentiator rules (P2).
    - New layers (L2–L6) are separate specifications, not extensions of L1.

    Tone: normative, one page. Do NOT reference or derive from Glama except the explicit "never Glama" prohibition in §4.
